## The constraints I set for myself

Before writing a line of code, I wrote down what the site had to be. The constraints turned out to matter more than the choices:

- **Lighthouse 95+ across all four categories, on real-world conditions, not just localhost.** Performance, Accessibility, Best Practices, SEO. No selective metric optimization.
- **First Contentful Paint under 1 second.** On a throttled 4G connection. Measured at the edge, not at localhost.
- **Cumulative Layout Shift below 0.05.** Reading shouldn't be interrupted by content jumping as the page loads.
- **Total JavaScript budget: 50KB per page, gzipped.** Most pages should ship near zero JS.
- **Mobile-first responsive at every breakpoint from 360px (small Android) to 1920px (4K).** No "best viewed in desktop" excuses.
- **WCAG AA contrast on every text element.** Including the decorative `~/` and `└` prefixes that screen readers should ignore.
- **Zero ongoing hosting cost up to 100,000 monthly visitors.** Cloudflare's free tier handles this; the architecture stays within those limits.
- **Single config file as the source of truth for branding.** Editing `siteConfig.name` or `siteConfig.author` updates the entire site. No grep-and-replace across 40 components.
- **Content lives in MDX files, version-controlled, no CMS.** Writing a post should mean opening a `.mdx` file in my editor. No dashboard. No round-trip.

Each constraint forced a downstream decision. The 50KB JS budget killed any React-based stack. The single-config rule forced extracting every hardcoded string from components. The Lighthouse 100 target meant self-hosting fonts to eliminate render-blocking external CSS.

Constraints aren't limits — they're directional. Mine pointed at Astro, Tailwind 4, and Cloudflare before I'd written a single line of code.

## Tech stack walkthrough

The stack:

- **[Astro 6](https://docs.astro.build/)** — static site generator. Renders content to HTML at build time. Ships zero JavaScript to the client by default. Components are `.astro` files with TypeScript script + HTML-like template syntax.
- **[Tailwind CSS 4](https://tailwindcss.com/)** — utility-first CSS framework, with the new Lightning CSS engine. Configured entirely in CSS (no `tailwind.config.js`). Tree-shaken automatically; only utilities actually referenced ship to production.
- **[MDX 5](https://mdxjs.com/) via `@astrojs/mdx`** — Markdown with optional embedded components. Used for all article content. Frontmatter typed via Zod through Astro's content collections.
- **[Shiki](https://shiki.style/)** — syntax highlighter built into Astro's MDX integration. Default theme: `github-dark`. Supports 200+ languages out of the box.
- **[Satori](https://github.com/vercel/satori) + [@resvg/resvg-js](https://github.com/yisibl/resvg-js)** — dynamic OG image generation at build time. Each article gets a 1200×630 PNG with the title and category.
- **[Pagefind](https://pagefind.app/)** — static full-text search. Index built from generated HTML; runs entirely client-side. No backend, no Algolia subscription.
- **[@fontsource](https://fontsource.org/) for JetBrains Mono + Inter Tight** — self-hosted fonts. Removed the render-blocking Google Fonts request and the privacy implications.
- **[Cloudflare Workers (static assets)](https://developers.cloudflare.com/workers/static-assets/)** — hosting via `wrangler.jsonc` configured for static-only deploy. No SSR. Free tier covers the foreseeable traffic ceiling.

On Cloudflare specifically: the first deploy failed because Cloudflare auto-detected the project as needing SSR and tried to install `@astrojs/cloudflare`, which then tried to bundle `node:fs` for the Satori OG endpoint and crashed. The fix was a one-file `wrangler.jsonc` declaring `[assets] directory = "./dist"` — explicit static-only deploy that bypasses the auto-detection. This pattern is documented in the [Cloudflare static assets docs](https://developers.cloudflare.com/workers/static-assets/) but takes some digging to find.

Astro vs the alternatives: Next.js ships React runtime even on fully-static pages, which violates the 50KB JS budget. SvelteKit is excellent but the ecosystem around Astro (integrations for sitemap, RSS, MDX) is more mature for content sites. Hugo is fast at builds but the Go templating syntax is awkward when you want component reuse. Astro is the choice you reach when you've tried the others and gotten tired of fighting them on content-focused work.

## Design decisions

The visual identity is what I'd call "terminal brutalist" — black background, neon green primary (#00FF66), monospace typography (JetBrains Mono), with a subtle CRT scanline overlay on desktop. The choices are connected:

**Pure black background (`#000000`) and neon green primary.** OLED screens render `#000` as pixels off, saving battery on mobile. The neon green passes WCAG AA contrast (16.1:1 against the body text color `#EDEDED`). The green is reserved for primary actions and accents; the body text is off-white, not green. Most readers don't realize the design is dramatic until they consciously look at it.

**JetBrains Mono variable font for everything.** Body text, headings, navigation, code blocks. The choice signals "this is a developer's space" before the first paragraph loads. JetBrains Mono is also one of the few mono fonts genuinely tuned for long-form reading — character ascenders are distinct at small sizes, and the variable-weight rendering means I can use one font for body (400 weight) and headings (700) without loading two separate font files.

**Inter Tight as the display font** for hero H1 and section H2. When the design needs a big punchy headline (`Ship serious software with <claude code/>`), Inter Tight is more visually grounded than mono at large sizes. Used sparingly so the mono identity stays dominant.

**Scanline overlay** via a pure-CSS `repeating-linear-gradient` on `body::after`. Zero JavaScript, `pointer-events: none`, disabled below 980px width to save battery on mobile. The effect is subtle by design — you notice it after a few minutes, not on first paint.

**ASCII grid background** on the hero section. Same technique: two crossed linear gradients masked with a radial fade. Adds visual depth without an image dependency. Disabled on mobile because the effect is less noticeable on small screens.

**Hover glows** using `box-shadow` with the green-glow CSS variable. Interactive elements feel alive without animation overhead. CSS-only; no JS event handlers.

**Three themes ship in the Pro version** (terminal, amber, cyan). All implemented via CSS variables under `[data-theme="..."]` selectors in `src/styles/themes.css`. Theme switching is one attribute change on `<html>`, persisted in localStorage. The inline script that reads localStorage runs before paint to avoid the flash-of-wrong-theme.

The whole design system fits in one file (`src/styles/global.css`, ~1700 lines) plus the theme tokens (`src/styles/themes.css`, ~100 lines). No CSS-in-JS, no Sass, no PostCSS plugins beyond Tailwind's own. Editing colors means editing CSS variables; the rest follows.

## Performance optimization

Getting to 100/100 Lighthouse wasn't one big optimization — it was the accumulation of a dozen small ones. The actual numbers from the latest desktop audit on the live production URL:

| Metric | Value |
|---|---|
| Performance | 100 |
| Accessibility | 95 (now 100 after the post-audit fixes below) |
| Best Practices | 100 |
| SEO | 100 |
| First Contentful Paint | 0.6s |
| Largest Contentful Paint | 0.6s |
| Total Blocking Time | 0ms |
| Cumulative Layout Shift | 0.002 |

How each one was earned:

**Zero JavaScript by default.** Astro's island architecture means components ship as plain HTML unless they explicitly opt into hydration via a `client:` directive. The blog has exactly two client-side scripts in production: the mobile menu toggle (~1KB) and the Pagefind search loader (lazy, only fetched when the search input is focused). Total runtime JS on a typical page: under 5KB.

**Self-hosted fonts via `@fontsource`.** The first audit flagged Google Fonts as render-blocking (293ms TTFB tax). Replacing the `<link>` to fonts.googleapis.com with `@import` statements in CSS pulled the font files into the same-origin bundle. CSS bundle grew ~70KB, but eliminated one network round-trip plus the Google privacy concern.

**Static OG images at build time.** Each article gets a unique 1200×630 PNG generated by Satori during the build, served as a regular static asset. No runtime image generation, no edge function cost, perfect cache-ability with `Cache-Control: public, max-age=31536000, immutable`.

**Critical CSS inlined automatically.** Astro inlines critical CSS into the HTML head for pages that need it. The rest of the CSS loads non-blocking. CLS stays near zero because the layout is computed from the initial HTML.

**ES2022 build target.** Vite's default `target: 'modules'` ships ES2020 with polyfills for older syntax. Bumping to ES2022 removes those polyfills and ships modern JS that any current browser handles natively. Typical bundle shrinks 8-15%.

**Image optimization via Sharp.** Astro processes images through Sharp at build time, generating WebP/AVIF variants and responsive `srcset` automatically. The few inline images on the site are tiny SVG icons; the rest of the visual weight is CSS gradients.

**Brotli compression at the edge.** Cloudflare serves all text assets (HTML, CSS, JS) with `Content-Encoding: br`. Brotli compresses about 15% better than gzip for JavaScript. Automatic on Cloudflare; no config required.

**Accessibility fixes specifically called out by Lighthouse:**

The first audit flagged contrast issues on the topbar and marquee text. The cause: `--text-dim: #666666` against the black background scored only 3.65:1 (WCAG AA requires 4.5:1 for normal text). Bumped to `#8a8a8a`, which scores 5.7:1. Done.

The audit also flagged heading order — the footer used `<h4>` elements after the homepage's `<h2>`, skipping `<h3>`. Renamed to `<h3>` and updated the corresponding CSS selector. Now the page heading hierarchy is `h1 → h2 → h3` cleanly.

The "Learn more" link without descriptive text was a false positive — that's the Astro dev toolbar overlay, only present in `npm run dev` mode, not in production. Same with the bf-cache warning (caused by Vite's hot-reload WebSocket, gone in prod).

The performance lesson: most "bad" scores come from running Lighthouse against `npm run dev` instead of `npm run preview`. Dev mode skips minification, runs the Vite hot-reload runtime, and includes the dev toolbar. Always audit against the production build.
